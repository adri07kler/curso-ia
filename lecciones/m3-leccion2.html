3.2 Qué información necesita la IA

Módulo 3 · Documento

Objetivo: Identificar la información mínima necesaria para que la IA genere respuestas útiles.

Introducción

Cuando trabajas con IA generativa, la calidad de la respuesta está directamente determinada por la calidad de la información que proporcionas. Dar información insuficiente produce respuestas genéricas. Dar información incorrecta produce respuestas peligrosas.

En esta lección aprenderás a identificar y estructurar los siete elementos esenciales que toda IA necesita para producir respuestas útiles y aplicables a tu proyecto.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Identificar los siete elementos esenciales de información para un contexto completo.
  • Distinguir entre información necesaria e información superflua.
  • Construir un contexto bien armado para cualquier proyecto de desarrollo.
  • Evaluar si un contexto proporcionado a la IA es suficiente para obtener una respuesta útil.

Información esencial

  1. Objetivo del proyecto: ¿Qué estamos construyendo y por qué?
  2. Rol esperado: ¿Qué rol debe adoptar la IA? (ej: "actúa como desarrollador senior de Python")
  3. Estado actual: ¿Qué existe ya? (código base, archivos, configuraciones)
  4. Restricciones técnicas: Stack tecnológico, versiones, limitaciones
  5. Lo que NO debe hacer: Límites explícitos
  6. Formato de salida: ¿Cómo quieres recibir la respuesta?
  7. Criterios de aceptación: ¿Cómo sabremos que la respuesta es correcta?

Ejemplo de contexto bien armado

Objetivo: Implementar autenticación con JWT
Rol: Desarrollador backend especializado en Node.js
Estado actual: Express configurado, BD PostgreSQL con tabla 'usuarios'
Stack: Node 20, Express 4, jsonwebtoken, bcrypt
Restricciones: No modificar package.json, no agregar dependencias
Formato: Código completo con comentarios mínimos

Desglose detallado de cada elemento

1. Objetivo del proyecto

Define el propósito general. No basta con decir "haz un login". La IA necesita entender el contexto de negocio: ¿es para una app interna? ¿un SaaS público? ¿una app móvil? El objetivo guía las decisiones técnicas que la IA tomará.

Ejemplo: "Sistema de reservas para un restaurante pequeño. Los clientes reservan online y el dueño confirma desde un panel."

2. Rol esperado

El rol condiciona el nivel de profundidad, el vocabulario y el estilo de la respuesta. Un "desarrollador junior" dará respuestas didácticas. Un "arquitecto de software" propondrá soluciones optimizadas para escalabilidad.

Ejemplo: "Actúa como desarrollador full-stack con 5 años de experiencia en React y Node.js."

3. Estado actual

La IA necesita saber qué existe ya para no sugerir cambios que rompan lo que funciona. Incluye archivos, configuraciones, dependencias instaladas y cualquier decisión ya tomada.

Ejemplo: "Ya tengo Express configurado con rutas básicas y una conexión a PostgreSQL usando Prisma ORM."

4. Restricciones técnicas

Cada equipo tiene reglas: versiones específicas, linters, convenciones de nombres, frameworks obligatorios. Sin estas restricciones, la IA puede generar código incompatible.

Ejemplo: "Usamos Node 20 LTS, Express 4, Prisma 5, TypeScript en strict mode. No usar clases, solo funciones."

5. Lo que NO debe hacer

Los límites evitan que la IA tome decisiones que no le corresponden.

Ejemplo: "No modifiques el archivo .env. No agregues dependencias externas. No cambies la estructura de la base de datos."

6. Formato de salida

Define cómo quieres recibir la respuesta: ¿código completo? ¿pseudocódigo? ¿explicación paso a paso? ¿solo el diff?

Ejemplo: "Quiero el código completo del archivo, con comentarios JSDoc en las funciones públicas."

7. Criterios de aceptación

Especifica cómo validarás que la respuesta es correcta.

Ejemplo: "El código debe pasar npm run lint sin errores, incluir tests unitarios con Jest, y cubrir el caso de token expirado."

Consejo del instructor: Si no estás seguro de qué información incluir, pregúntate: "Si un desarrollador nuevo se incorporara al proyecto, ¿qué necesitaría saber para hacer esta tarea?" Todo eso es contexto relevante.

Errores frecuentes

  • Asumir que la IA "sabe" el proyecto: Cada sesión de chat es independiente. La IA no recuerda conversaciones anteriores a menos que se lo indiques explícitamente.
  • Dar solo restricciones técnicas sin objetivo: Decir "React 18, Node 20" sin explicar el propósito hace que la IA no pueda priorizar decisiones de diseño.
  • Olvidar los criterios de aceptación: Sin ellos, la IA no sabe cuándo su respuesta es suficiente y puede entregar trabajo a medias.

Buenas prácticas

  • Usa siempre los siete elementos esenciales como checklist antes de enviar un contexto.
  • Mantén un archivo "context-packet.md" en la raíz de cada proyecto.
  • Actualiza el contexto cuando cambien las versiones o restricciones del proyecto.
  • Incluye ejemplos concretos de lo que esperas como salida.
  • Sé específico con las restricciones: "Node 20 LTS" es mejor que "Node reciente".

Actividad práctica

Selecciona un proyecto real o ficticio y completa los siete elementos esenciales:

  1. Objetivo del proyecto
  2. Rol esperado
  3. Estado actual
  4. Restricciones técnicas
  5. Lo que NO debe hacer
  6. Formato de salida
  7. Criterios de aceptación

Mini reto

Escribe el contexto para construir una API de clima. Debes incluir los siete elementos. Luego pídele a una IA que genere el código usando solo tu contexto. ¿Fue suficiente? Si la IA pidió más información, identifica qué elemento faltó o estuvo incompleto.

Resumen

  • La IA necesita siete elementos esenciales: objetivo, rol, estado actual, restricciones, límites, formato y criterios de aceptación.
  • La calidad de la información que proporcionas determina directamente la calidad de la respuesta.
  • Un contexto completo reduce iteraciones, ahorra tokens y minimiza alucinaciones.
  • Siete elementos pueden expresarse en 5-10 líneas; no necesitas documentos extensos.
