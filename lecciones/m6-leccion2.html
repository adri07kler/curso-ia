6.2 Convertir la idea en un requerimiento

Módulo 6 · Ejercicio

Objetivo: Aplicar las técnicas del Módulo 2 para refinar la idea en un requerimiento completo.

Introducción

Una idea inicial es como un boceto en servilleta: captura la esencia pero carece del detalle necesario para construir. En esta lección tomarás la idea del proyecto final y la transformarás en un requerimiento formal utilizando las técnicas de descubrimiento, refinamiento y especificación que aprendiste en el Módulo 2.

En proyectos reales, esta traducción de "idea" a "requerimiento" es donde fracasan la mayoría de los proyectos. Un requerimiento ambiguo o incompleto genera código incorrecto, retrabajo y frustración. Con IA, el problema se magnifica: la IA no puede preguntar "¿qué quisiste decir aquí?" — simplemente generará algo que parece correcto pero no lo es.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Aplicar preguntas de descubrimiento para desambiguar una idea inicial
  • Redactar requerimientos funcionales claros y verificables
  • Escribir criterios de aceptación usando el formato Given-When-Then
  • Identificar casos borde relevantes para el proyecto
  • Definir explícitamente qué queda fuera del alcance del proyecto

Desarrollo del contenido

De la idea al requerimiento: el puente necesario

La idea inicial del proyecto final es:

"Necesitamos una aplicación web para gestionar tareas personales (to-do list) pero que permita a los usuarios organizar tareas en proyectos, asignar etiquetas, establecer prioridades y fechas de vencimiento. También debe tener recordatorios y un dashboard con estadísticas básicas."

Para convertirla en un requerimiento, necesitas aplicar un proceso estructurado de refinamiento. Cada técnica del Módulo 2 aborda una dimensión diferente:

Paso 1: Preguntas de descubrimiento (Lección 2.2)

Las preguntas de descubrimiento revelan información que la idea inicial no incluye. Para nuestro proyecto, algunas preguntas clave son:

  • ¿Quién puede crear proyectos? ¿Cualquier usuario o solo administradores?
  • ¿Las tareas pueden tener múltiples etiquetas o solo una?
  • ¿Los recordatorios son por email, notificación en navegador, o ambos?
  • ¿El dashboard debe mostrar datos en tiempo real o con datos históricos?
  • ¿Los usuarios necesitan registrarse o puede usar el sistema sin cuenta?

Paso 2: Requerimientos funcionales (Lección 2.3)

Un requerimiento funcional describe qué debe hacer el sistema. Debe ser específico, medible y verificable. Ejemplo:

RF-01: El sistema permitirá a los usuarios registrados crear proyectos con nombre, descripción y color identificativo.
RF-02: El sistema permitirá crear tareas dentro de un proyecto con título, descripción, prioridad (alta/media/baja), fecha de vencimiento y etiquetas.
RF-03: El sistema enviará un recordatorio por notificación del navegador 24 horas antes de la fecha de vencimiento de una tarea.

Paso 3: Criterios de aceptación Given-When-Then (Lección 2.4)

Los criterios de aceptación definen cuándo un requerimiento se considera cumplido. Usando el formato Given-When-Then:

RF-02: Crear tarea con prioridad
Given un usuario autenticado en un proyecto existente
When el usuario crea una tarea con prioridad "alta", título "Revisar diseño" y fecha de vencimiento "2026-08-15"
Then la tarea se guarda con esos datos
And aparece en el listado de tareas del proyecto con la etiqueta "Prioridad: Alta" resaltada en rojo

Paso 4: Casos borde (Lección 2.5)

Los casos borde son situaciones que la idea inicial no contempla. Para nuestro proyecto:

  • ¿Qué pasa si un usuario crea una tarea sin fecha de vencimiento?
  • ¿Qué ocurre si se elimina un proyecto que contiene tareas?
  • ¿Cómo se comporta el dashboard si no hay tareas registradas?
  • ¿Qué sucede si el usuario programa un recordatorio para una fecha ya pasada?

Paso 5: Fuera de alcance

Definir qué NO hará el proyecto es tan importante como definir qué hará. Para nuestro proyecto:

🔴 Fuera de alcance:
- Colaboración en tiempo real (varios usuarios editando la misma tarea)
- Aplicación móvil nativa (solo versión web responsive)
- Integración con calendarios externos (Google Calendar, Outlook)
- Notificaciones por correo electrónico (solo notificaciones en navegador)
- Exportación a PDF o CSV

Ejemplos de refinamiento en otros contextos

  • Automatización: "Descargar facturas del correo" → "RF-01: El script se conectará a IMAP cada hora, buscará emails con asunto 'Factura' en las últimas 24h, descargará los adjuntos PDF y los guardará en /facturas/{año}/{mes}/"
  • IA generativa: "Generar descripciones de productos" → "RF-02: Dado un nombre de producto, categoría y 3 palabras clave, el sistema generará una descripción de 50-100 palabras usando IA, en tono profesional y sin markdown"
  • Prompt engineering: "Optimizar prompts largos" → "RF-03: El sistema recibirá un prompt de entrada y devolverá una versión optimizada que conserve el significado pero reduzca el número de tokens en al menos un 30%"
💡 Consejo del instructor: Cuando trabajes con IA, un requerimiento ambiguo es más peligroso que un requerimiento incompleto. La IA no te preguntará "¿esto qué significa?" — generará código con su propia interpretación, que casi siempre será incorrecta. Si no puedes escribir un criterio Given-When-Then para una funcionalidad, es porque no la entiendes lo suficiente como para delegarla a una IA.

Errores frecuentes

  • Confundir "requerimiento" con "historia de usuario": Una historia de usuario es una promesa de conversación; un requerimiento es una especificación verificable
  • Escribir requerimientos ambiguos: Usar palabras como "rápido", "moderno", "intuitivo" sin definirlas
  • Olvidar los casos borde: Solo especificar el camino feliz y asumir que todo lo demás no pasará
  • No definir el fuera de alcance: El cliente asumirá que todo está incluido si no se dice explícitamente lo contrario
  • Requerimientos no verificables: "El sistema debe ser fácil de usar" no se puede probar; "el 90% de los usuarios debe completar el registro en menos de 2 minutos" sí

Buenas prácticas

  • Usar numeración única: RF-01, RF-02... para trazabilidad
  • Escribir un criterio de aceptación por requerimiento: Si tiene múltiples, divídelo
  • Revisar requerimientos con otra persona: El sesgo del autor oculta ambigüedades
  • Mantener un glosario de términos: "Usuario", "Administrador", "Tarea" deben tener definiciones únicas
  • Versionar los requerimientos: Los requerimientos cambian; el historial de cambios es documentación valiosa

Actividad práctica

Aplica las técnicas del Módulo 2 para refinar la idea del proyecto final en un requerimiento completo:

  1. Aplica las preguntas de descubrimiento (Lección 2.2)
  2. Define requerimientos funcionales claros (Lección 2.3)
  3. Escribe criterios de aceptación usando Given-When-Then (Lección 2.4)
  4. Identifica casos borde (Lección 2.5)
  5. Define qué queda fuera de alcance

Mini reto

Toma la idea que registraste en la lección 6.1 (o pide una nueva a la IA) y aplica el proceso completo de refinamiento: preguntas de descubrimiento, requerimientos funcionales, criterios Given-When-Then, casos borde y fuera de alcance. Compara tu resultado con el de un compañero o con una segunda iteración de la IA para identificar diferencias.

Resumen

  • La idea inicial debe pasar por un proceso de refinamiento estructurado antes de usarse con IA
  • Las preguntas de descubrimiento revelan información implícita en la idea original
  • Los requerimientos funcionales deben ser específicos, medibles y verificables
  • Los criterios Given-When-Then eliminan la ambigüedad en la interpretación de requerimientos
  • Los casos borde y el fuera de alcance evitan sorpresas durante el desarrollo
  • Un requerimiento bien escrito es la base para un contexto IA efectivo
