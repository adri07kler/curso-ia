6.6 Validar la solución

Módulo 6 · Ejercicio

Objetivo: Aplicar validación y pruebas a la solución generada.

Introducción

El código generado por IA no es el producto final — es un borrador que debe pasar por un proceso de validación riguroso. En esta lección aplicarás las técnicas de validación del Módulo 4 (checklist de revisión, pruebas mínimas, seguridad) al código generado en la lección anterior.

En proyectos reales, saltarse la validación de código generado por IA es la principal causa de bugs en producción. Un código que "se ve bien" puede contener vulnerabilidades graves, errores lógicos sutiles o simplemente no cumplir con los requerimientos.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Aplicar una checklist de revisión sistemática al código generado por IA
  • Escribir y ejecutar pruebas mínimas (humo, caso feliz, caso borde, error)
  • Identificar problemas de seguridad comunes en código generado por IA
  • Clasificar la solución como aprobada, requiere cambios o rechazada
  • Documentar los cambios realizados sobre el código generado

Desarrollo del contenido

Checklist de revisión del Módulo 4

La checklist de validación cubre las siguientes dimensiones. Aplica cada una al código generado:

1. Correctitud funcional

  • ¿El código implementa exactamente lo que pide el requerimiento?
  • ¿Maneja los casos borde identificados?
  • ¿Las funciones producen el resultado esperado para entradas válidas?
  • ¿El flujo de navegación entre pantallas es correcto?

2. Calidad del código

  • ¿Los nombres de variables y funciones son descriptivos?
  • ¿El código sigue las convenciones del lenguaje?
  • ¿Hay código muerto, duplicado o redundante?
  • ¿La estructura es modular y mantenible?

3. Seguridad

  • ¿Se validan las entradas del usuario?
  • ¿Hay riesgo de XSS (datos renderizados sin sanitizar)?
  • ¿Se usan datos directamente del localStorage sin validación?
  • ¿Hay exposición de información sensible?

4. Rendimiento y usabilidad

  • ¿Las operaciones son eficientes para el volumen de datos esperado?
  • ¿La interfaz es responsive?
  • ¿Los mensajes de error son claros para el usuario?
  • ¿La experiencia de carga es aceptable?

Tipos de pruebas mínimas

Para cada función generada por IA, debes ejecutar al menos:

  • Prueba de humo: La función se ejecuta sin errores con datos válidos
  • Prueba de caso feliz: Con datos típicos, produce el resultado esperado
  • Prueba de caso borde: Con datos límite (vacío, nulo, extremo) se comporta adecuadamente
  • Prueba de error: Con datos inválidos, lanza un error claro o maneja la situación

Decisión final

Al finalizar la validación, clasifica la solución en una de tres categorías:

✅ Aprobada: Cumple todos los criterios de la checklist y pasa todas las pruebas. Se integra a producción.
🔧 Requiere cambios: Tiene problemas menores o falta de casos borde. Se documentan los cambios necesarios y se corrige.
❌ Rechazada: No cumple requerimientos fundamentales, tiene vulnerabilidades graves o errores estructurales. No se integra.

Ejemplos de problemas detectados en validación

  • Desarrollo web: Una función de búsqueda que filtra en cliente pero debería filtrar también en datos añadidos dinámicamente — caso borde no contemplado
  • Seguridad: Un campo de texto que renderiza HTML directamente (innerHTML) sin sanitizar — riesgo de XSS
  • Automatización: Un script que elimina archivos sin confirmación previa — riesgo de pérdida de datos
  • IA generativa: Una función que usa un API key hardcodeado en el código — vulnerabilidad crítica
💡 Consejo del instructor: La validación de código generado por IA requiere un cambio de mentalidad. Normalmente revisamos código asumiendo que está bien hasta que encontramos algo mal. Con código de IA, asume que está mal hasta que demuestres que está bien. Este enfoque (default skeptic) te ahorrará horas de depuración.

Errores frecuentes

  • Probar solo el caso feliz: El camino más común funciona, pero todo lo demás falla
  • Confiar en que la IA "sabe" de seguridad: La IA replica vulnerabilidades de sus datos de entrenamiento
  • No probar en el entorno real: El código funciona en el chat de la IA pero no en el navegador real
  • Aceptar código sin ejecutarlo: "Se ve bien" no es una validación válida
  • Documentar solo los errores: También debes documentar qué funciona y por qué lo aceptaste

Buenas prácticas

  • Siempre probar en el entorno real: El chat de la IA no es un runtime
  • Mantener una checklist impresa o digital: No confíes en la memoria para cubrir todas las dimensiones
  • Probar con datos reales: Los datos de prueba deben reflejar el uso en producción
  • Revisar seguridad antes que funcionalidad: Un código seguro que falla es mejor que uno vulnerable que funciona
  • Documentar cada hallazgo: Para trazabilidad y mejora de prompts futuros

Actividad práctica

  1. Revisa el código usando la checklist del Módulo 4
  2. Escribe y ejecuta pruebas mínimas
  3. Identifica problemas de seguridad
  4. Decide: aprobada, requiere cambios, o rechazada
  5. Documenta los cambios sobre el código generado

Mini reto

Pide a la IA que genere intencionalmente código con errores (por ejemplo, "genera una función de validación de email con al menos 3 vulnerabilidades"). Luego aplica la checklist de validación para identificar cada problema. Este ejercicio inverso entrena tu ojo para detectar errores comunes de IA. Documenta los 3 errores que encontraste y cómo los corregirías.

Resumen

  • Todo código generado por IA debe pasar por una validación sistemática antes de integrarse
  • La checklist de revisión cubre: correctitud, calidad, seguridad y usabilidad
  • Las pruebas mínimas son: humo, caso feliz, caso borde y error
  • La clasificación final puede ser: aprobada, requiere cambios o rechazada
  • Asume que el código está mal hasta que demuestres lo contrario (default skeptic)
  • Documentar cada hallazgo mejora los prompts futuros y la trazabilidad del proyecto

Entrega: Reporte de validación + cambios realizados + pruebas ejecutadas.
