5.4 Revisión de código generado

Módulo 5 · Video

Objetivo: Aprender técnicas de code review específicas para código generado por IA.

Introducción

El code review tradicional entre humanos asume que el autor tenía una intención y aplicó criterio al escribir cada línea. Con código generado por IA, no hay intención — solo patrones estadísticos. Esto exige un enfoque de revisión diferente, más metódico y con énfasis en áreas donde la IA falla sistemáticamente. Esta lección te dará un proceso de 4 pasadas para revisar código de IA de forma eficiente y completa.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Aplicar un proceso de revisión de código de 4 pasadas específico para código generado por IA
  • Identificar señales de alerta comunes en código generado por IA
  • Detectar código muerto, dependencias ficticias y funciones comodín
  • Evaluar si el código generado cumple con los estándares del proyecto

Proceso de revisión

  1. Primera pasada — Estructural: ¿Sigue los patrones del proyecto? ¿Respeta convenciones?
  2. Segunda pasada — Lógica: ¿La lógica es correcta? ¿Faltan condiciones? ¿Los bucles terminan?
  3. Tercera pasada — Seguridad: ¿Validación de entrada? ¿Sanitización? ¿Credenciales?
  4. Cuarta pasada — Mantenibilidad: ¿Será fácil de modificar? ¿Funciones con una sola responsabilidad?

Primera pasada — Estructural

Verifica que el código siga las convenciones del proyecto: nombres de variables en el idioma correcto, indentación consistente, estructura de archivos según el estándar del equipo. La IA mezcla estilos porque se entrena con código de miles de fuentes diferentes.

Segunda pasada — Lógica

Traza mentalmente la ejecución con diferentes entradas. ¿Los bucles tienen condición de término? ¿Las recursiones tienen caso base? ¿Los condicionales cubren todos los casos? Esta es la pasada donde más bugs encuentra la IA.

Tercera pasada — Seguridad

Busca vulnerabilidades específicas: entradas de usuario sin validar, consultas SQL concatenadas, credenciales hardcodeadas, falta de rate limiting, tokens expuestos. La IA no diferencia entre código seguro e inseguro.

Cuarta pasada — Mantenibilidad

Evalúa si el código será fácil de mantener en 6 meses: funciones con una sola responsabilidad, nombres descriptivos, ausencia de números mágicos, comentarios útiles (no genéricos). La IA genera código funcional pero rara vez mantenible.

Señales de alerta específicas de IA

  • Funciones "comodín" que hacen demasiado
  • Código muerto (variables no usadas)
  • Dependencias que no existen
  • Código que parece de otro lenguaje
  • Comentarios genéricos ("# Add your code here")
  • Importaciones redundantes o incorrectas
  • Funciones sin docstring o con documentación genérica
  • Uso de librerías obsoletas o deprecadas
  • Manejo de errores ausente o genérico (except: pass)
  • Código duplicado (la IA repite patrones innecesariamente)

Aplica las 4 pasadas en orden. No saltes ninguna — cada una detecta un tipo distinto de problema.

Consejo del instructor: Crea una checklist física (papel o archivo) con las 4 pasadas y las señales de alerta. Revisa cada función generada por IA contra esa checklist antes de integrarla. En equipos, incluye la revisión de código IA como un paso obligatorio en el proceso de merge request.

Errores frecuentes

  • Revisar solo la primera pasada: Detectar problemas de estilo es fácil; los errores graves están en la lógica y seguridad
  • Confiar en que "la IA ya lo revisó": La IA no revisa su propio código; cada respuesta es independiente
  • No probar después de revisar: La revisión manual encuentra problemas, pero las pruebas encuentran los que pasaron desapercibidos
  • Aprobar código que no se entiende completamente: Si no entiendes cada línea, no deberías aprobarlo

Buenas prácticas

  • Siempre las 4 pasadas sin saltarse ninguna: Cada pasada tiene un propósito específico
  • Usar linters y análisis estático: ESLint, Pylint, SonarQube para detectar problemas automáticos
  • Ejecutar las pruebas antes de revisar: Si las pruebas fallan, ni siquiera empieces la revisión manual
  • Documentar cada cambio aceptado o rechazado: Lleva un registro de qué modificaste y por qué
  • Revisar en pares cuando sea posible: Dos pares ven más que uno, especialmente con código de IA

Actividad práctica

Revisa el siguiente código generado por IA usando las 4 pasadas:

def process_data(data):
    result = []
    for i in data:
        # TODO: implement
        pass
    return result

def get_user(id):
    import sqlite3
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    cursor.execute(f"SELECT * FROM users WHERE id = {id}")
    return cursor.fetchone()

def calculate(x, y):
    z = x + y
    return z
  1. Pasada estructural: ¿qué problemas ves?
  2. Pasada lógica: ¿qué errores de lógica hay?
  3. Pasada seguridad: ¿qué vulnerabilidades encuentras?
  4. Pasada mantenibilidad: ¿qué dificultaría el mantenimiento?
Solución: process_data está incompleta (pass). get_user tiene SQL injection (f-string). calculate mezcla suma con nombre "calculate". La conexión SQL no se cierra. No hay manejo de errores en ninguna función.

Mini reto

Pide a una IA que genere una función para autenticar usuarios con email y contraseña. Aplica las 4 pasadas de revisión. Documenta al menos 3 problemas que encuentres en cada pasada. Luego compara: ¿cuántos problemas encontró la revisión humana vs. los que encontraría un scanner automático?

Resumen

  • El proceso de 4 pasadas cubre: estructura, lógica, seguridad y mantenibilidad
  • La IA tiene señales de alerta específicas: dependencias falsas, código muerto, funciones comodín
  • La revisión de código IA requiere más atención que la revisión humana
  • Usar linters y análisis estático complementa la revisión manual
  • No aprobar código que no se entiende completamente
