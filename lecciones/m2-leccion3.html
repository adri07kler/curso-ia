2.3 Requerimientos funcionales

Módulo 2 · Documento

Objetivo: Aprender a escribir requerimientos funcionales claros y accionables.

Características de un buen requerimiento funcional

  • Claro: Una sola interpretación posible
  • Verificable: Se puede probar (pasa/no pasa)
  • Completo: Incluye toda la información necesaria
  • Consistente: No contradice otros requerimientos
  • Necesario: Realmente se necesita

Ejemplo bien escrito

RF-001: El sistema debe permitir que un usuario registrado inicie sesión
con su correo electrónico y contraseña.
- El campo de correo debe aceptar formatos estándar (ej: usuario@dominio.com)
- La contraseña debe tener al menos 8 caracteres
- Tras 3 intentos fallidos, la cuenta debe bloquearse por 30 minutos
- El sistema debe devolver un mensaje de error específico para cada tipo de fallo

Introducción

Los requerimientos funcionales son la columna vertebral de todo proyecto de software. Definen qué debe hacer el sistema: cada función, cada regla de negocio, cada interacción esperada. En esta lección aprenderás a redactar requerimientos funcionales con la claridad y precisión necesarias para que un equipo de desarrollo (humano o IA) los entienda sin ambigüedad. Dominar esta habilidad es clave para reducir iteraciones, evitar malentendidos y entregar software que realmente resuelva el problema planteado.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Comprender la diferencia entre requerimientos funcionales y no funcionales
  • Identificar las características de un requerimiento funcional bien redactado
  • Redactar requerimientos funcionales claros, verificables y completos
  • Aplicar el formato estándar de identificación RF-XXX para documentar requerimientos
  • Analizar requerimientos vagos y transformarlos en especificaciones accionables
  • Evaluar la calidad de un conjunto de requerimientos funcionales

Desarrollo: Cómo escribir requerimientos funcionales de calidad

¿Qué es un requerimiento funcional?

Un requerimiento funcional describe una capacidad o comportamiento específico que el sistema debe tener. Responde a la pregunta: "¿Qué debe hacer el sistema?". Ejemplos: "el sistema debe enviar un correo de bienvenida", "el sistema debe calcular el total del carrito", "el sistema debe permitir filtrar productos por categoría".

Identificación y nomenclatura

Cada requerimiento funcional debe tener un identificador único. El formato más común es RF-XXX (Requerimiento Funcional + número secuencial). Esto permite trazabilidad: en pruebas, en código, en documentación. Un RF-001 en la especificación debe corresponder a una prueba RF-001-TEST y a un commit que haga referencia a RF-001.

Estructura de un requerimiento funcional

Un requerimiento funcional bien escrito incluye: (1) identificador único, (2) título descriptivo, (3) descripción detallada del comportamiento esperado, (4) condiciones y restricciones, (5) criterios de aceptación implícitos o explícitos. No debe incluir detalles de implementación: cómo se implementa es decisión del equipo técnico; qué debe hacer es responsabilidad del requerimiento.

Requerimientos funcionales vs no funcionales

Mientras los funcionales describen comportamientos, los no funcionales describen cualidades del sistema: rendimiento (<500ms), seguridad (HTTPS, bcrypt), escalabilidad (soportar 10,000 usuarios concurrentes), disponibilidad (99.9% uptime). Ambos tipos son necesarios, pero esta lección se enfoca en los funcionales.

Ejemplos adicionales

RF-002 — Registro de usuario:
El sistema debe permitir que un visitante se registre proporcionando: nombre completo, correo electrónico válido y una contraseña de al menos 8 caracteres.
- El sistema debe enviar un correo de verificación al registro exitoso
- El sistema no debe permitir registros con correos duplicados
- El sistema debe mostrar "Correo ya registrado" si el correo existe
RF-003 — Búsqueda de productos:
El sistema debe permitir buscar productos por nombre, categoría y rango de precio.
- Los resultados deben mostrar nombre, precio, imagen y calificación
- La búsqueda debe devolver resultados en menos de 2 segundos
- Si no hay resultados, mostrar "No se encontraron productos"

💡 Consejo del instructor

💡 Consejo profesional: Al escribir requerimientos funcionales, usa siempre el verbo "debe" (must en inglés). Esto indica obligación. Frases como "puede", "sería bueno" o "idealmente" generan ambigüedad. Un requerimiento funcional no es negociable: o se cumple o no se cumple. Si algo es opcional, documéntalo como "deseable" en una sección aparte, no como RF.

⚠ Errores frecuentes

  • Mezclar implementación con especificación: Escribir "el sistema debe usar una base de datos MongoDB" en lugar de "el sistema debe almacenar usuarios y productos". La IA o el equipo decidirán la tecnología.
  • Requerimientos imposibles de verificar: Frases como "el sistema debe ser intuitivo" son subjetivas. En su lugar: "el sistema debe permitir completar el registro en menos de 3 pasos".
  • Requerimientos demasiado grandes: Un RF que describe un flujo completo de 10 pasos es difícil de probar. Divide en RF más pequeños y específicos.
  • Olvidar condiciones de error: Solo describir el camino feliz. Un buen RF incluye qué pasa cuando algo falla.
  • Escribir para ti mismo: Usar jerga interna o sobrentender conocimiento del negocio. Un RF debe ser comprensible para cualquier persona del equipo, incluyendo nuevos integrantes.

✔ Buenas prácticas

  • Usa numeración única: Identifica cada RF con un código como RF-001, RF-002. Facilita trazabilidad en pruebas y desarrollo.
  • Escribe en voz activa: "El sistema debe mostrar" en lugar de "debe ser mostrado por el sistema". Es más directo y claro.
  • Incluye ejemplos concretos: Un RF sobre formatos de fecha debe decir "dd/mm/aaaa" no "formato estándar de fecha".
  • Revisa con pares: Pide a otro miembro del equipo que lea el RF y lo interprete. Si interpreta algo diferente, tu RF no es claro.
  • Mantén un glosario: Si usas términos del negocio, defínelos. "Cliente premium", "orden cancelada", "inventario disponible" pueden significar cosas distintas para diferentes personas.

Actividad práctica

Imagina que trabajas en un equipo que desarrolla una aplicación de delivery de comida. Redacta 5 requerimientos funcionales para la funcionalidad de "seguimiento de pedido en tiempo real". Cada RF debe incluir: identificador, título, descripción detallada y condiciones de error. Aplica las características de claridad, verificabilidad, completitud, consistencia y necesidad que viste en esta lección.

Mini reto

Toma un requerimiento vago como "El sistema debe tener un buen sistema de notificaciones" y transfórmalo en al menos 4 requerimientos funcionales específicos y verificables. Incluye notificaciones push, email, SMS e in-app. Define para cada uno: cuándo se dispara, qué contenido muestra, y qué ocurre si el canal de notificación falla.

Resumen

  • Los requerimientos funcionales definen qué debe hacer el sistema, no cómo.
  • Deben ser claros, verificables, completos, consistentes y necesarios.
  • Usa identificadores RF-XXX para trazabilidad entre requerimientos, pruebas y código.
  • Incluye siempre condiciones de error y caminos alternativos.
  • Separa requerimientos funcionales de no funcionales: comportamientos vs cualidades.
  • Un RF bien escrito reduce iteraciones, evita malentendidos y acelera el desarrollo.

Conclusión: La calidad de tus requerimientos funcionales determina la calidad del software que recibirás. Invertir tiempo en escribirlos bien es la forma más efectiva de ahorrar tiempo en desarrollo, pruebas y correcciones. Un requerimiento claro es el mejor regalo que puedes darle a tu equipo.
