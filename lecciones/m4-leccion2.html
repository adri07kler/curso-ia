4.2 Qué no debe hacerse: pedir "todo el sistema"

Módulo 4 · Documento

Objetivo: Entender por qué pedir el sistema completo a la IA es un error.

Por qué es mala idea

  • La IA pierde el contexto global y genera inconsistencias
  • Las partes del sistema se acoplan mal entre sí
  • Es imposible revisar todo el código generado de una vez
  • Los errores son difíciles de localizar
  • El resultado suele ser código genérico que no se adapta a tus necesidades

Mal ejemplo

"Genera un sistema completo de e-commerce con autenticación, carrito, pagos, y panel de administración"

Bien ejemplo (encadenado)

Paso 1: "Genera solo el modelo de datos para usuarios y productos"
Paso 2: "Genera el endpoint de registro de usuarios"
Paso 3: "Genera el endpoint de listar productos"
Regla: Si el prompt tiene más de 3 requerimientos distintos, divídelo.

Introducción

Una de las tentaciones más comunes al comenzar con desarrollo asistido por IA es pedir todo de una vez: "Genera un sistema completo". En esta lección entenderás por qué este enfoque está condenado al fracaso y aprenderás a dividir tus solicitudes en fragmentos manejables que la IA pueda procesar correctamente.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Identificar por qué pedir el sistema completo a la IA genera inconsistencias y errores.
  • Diferenciar entre un prompt monolítico y uno encadenado.
  • Aplicar la regla de máximo 3 requerimientos por prompt.
  • Diseñar una secuencia de prompts que construyan un sistema paso a paso.
  • Reconocer cuándo un prompt es demasiado grande y requiere división.

Desarrollo del contenido

El problema del contexto limitado

Los modelos de lenguaje tienen una ventana de contexto finita (medida en tokens). Cuando pides un sistema completo, la IA debe distribuir su atención entre múltiples componentes, lo que resulta en código genérico, inconsistente y con acoplamientos frágiles. Además, al generar todo de una vez, no puedes verificar cada parte antes de continuar.

Las consecuencias de los prompts monolíticos

  • Inconsistencias internas: La IA puede nombrar una variable "usuario" en un módulo y "user" en otro.
  • Acoplamiento frágil: Las interfaces entre módulos no se comunican correctamente.
  • Depuración imposible: Si algo falla, no sabes si el error está en la autenticación, el carrito o los pagos.
  • Código genérico: La IA recurre a soluciones estándar que no se adaptan a tu arquitectura específica.
  • Revisión humana inviable: Revisar 500 líneas de código generadas en un solo prompt es propenso a errores.

La técnica de encadenamiento de prompts

El encadenamiento consiste en dividir un sistema grande en una secuencia de prompts pequeños, donde el resultado de cada paso sirve como contexto para el siguiente. Esto permite mantener coherencia, revisar cada fragmento y corregir sobre la marcha.

Ejemplos

Ejemplo 1: Sistema de gestión de tareas (To-Do List)

Prompt monolítico (malo):

"Crea un sistema completo de gestión de tareas con usuarios, autenticación, CRUD de tareas, categorías, y panel de administración."

Encadenamiento (bueno):

Paso 1: "Crea el modelo de datos SQL para un sistema de tareas. Tablas: usuarios, tareas, categorías. Incluye tipos de datos y claves foráneas."
Paso 2: "Basado en este modelo, genera el endpoint POST /api/register para registro de usuarios con validación de email y contraseña."
Paso 3: "Ahora genera el endpoint GET /api/tareas que devuelva las tareas del usuario autenticado, ordenadas por fecha."

Ejemplo 2: API REST para e-commerce

Enfoque por capas:

PasoPromptResultado esperado
1"Define el esquema de base de datos para productos, carrito y órdenes"Modelo de datos
2"Genera el endpoint GET /api/productos con paginación y filtros"Endpoint funcional
3"Añade autenticación JWT al endpoint anterior"Seguridad integrada
4"Crea pruebas unitarias para el endpoint de productos"Tests automatizados

💡 Consejo del instructor: Crea un archivo "contexto.md" en tu proyecto donde vayas acumulando las decisiones y especificaciones. Al iniciar un nuevo prompt, copia las partes relevantes del contexto. Esto ayuda a la IA a mantener coherencia entre pasos.
⚠ Errores comunes:
  • No revisar entre pasos: Acumular errores que se multiplican.
  • Cambiar de tecnología a mitad del encadenamiento: Pedir Express en un paso y FastAPI en el siguiente.
  • No proporcionar contexto suficiente: Asumir que la IA recuerda lo que generó hace varios pasos sin reafirmarlo.

Buenas prácticas

  • ✔ Define el alcance de cada prompt antes de escribirlo.
  • ✔ Incluye el resultado del paso anterior como contexto del siguiente.
  • ✔ Usa nomenclatura coherente en todos los pasos.
  • ✔ Revisa y prueba cada paso antes de continuar.
  • ✔ Si un paso genera más de 100 líneas, considera dividirlo.
  • ✔ Documenta la secuencia de prompts para poder reproducirla.

Actividad práctica

Toma un sistema simple (ej: un blog con posts, comentarios y usuarios) y diseña una secuencia de mínimo 5 prompts encadenados que construyan el sistema paso a paso. Para cada prompt, especifica: objetivo, qué contexto necesita del paso anterior, y qué resultado esperas.

Mini reto

Refactoriza un prompt monolítico que encuentres en internet o que hayas usado antes. Divídelo siguiendo la técnica de encadenamiento y documenta cómo mejora la calidad del resultado. Compara el antes y después.

Resumen

  • Pedir el sistema completo a la IA genera código inconsistente y difícil de depurar.
  • La ventana de contexto limitada hace que la IA pierda coherencia en prompts grandes.
  • El encadenamiento de prompts permite construir sistemas complejos paso a paso.
  • La regla de oro: máximo 3 requerimientos por prompt.
  • Cada paso del encadenamiento es revisable, testeable y corregible de forma independiente.
