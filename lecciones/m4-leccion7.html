4.7 Decidir si aceptar o rechazar código

Módulo 4 · Quiz

Objetivo: Evaluar la capacidad de decidir si un fragmento de código generado por IA es aceptable.

Introducción

Esta evaluación final del módulo 4 pondrá a prueba tu capacidad para tomar decisiones críticas sobre código generado por IA. No se trata solo de saber si el código funciona, sino de evaluar aspectos como seguridad, mantenibilidad, rendimiento y responsabilidad. Las preguntas están diseñadas para evaluar tu criterio como ingeniero, no tu memoria.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Evaluar críticamente si el código generado por IA cumple con los estándares de calidad necesarios.
  • Diferenciar entre código aceptable, código que requiere correcciones y código que debe rechazarse.
  • Justificar las decisiones de aceptación o rechazo basándose en criterios objetivos (seguridad, rendimiento, mantenibilidad).
  • Identificar la responsabilidad del ingeniero en el ciclo de desarrollo asistido por IA.

Desarrollo del contenido

Criterios para decidir aceptar o rechazar

Antes de responder el quiz, repasemos los criterios fundamentales que determinan si un fragmento de código generado por IA es aceptable:

  • Funcionalidad correcta: El código debe hacer exactamente lo que se solicitó, ni más ni menos.
  • Seguridad: No debe introducir vulnerabilidades (inyección, exposición de datos, etc.).
  • Eficiencia: La complejidad algorítmica debe ser adecuada al contexto de uso.
  • Mantenibilidad: El código debe ser legible, estar bien estructurado y seguir las convenciones del proyecto.
  • Dependencias: Solo debe utilizar las dependencias necesarias y aprobadas por el equipo.
  • Pruebas: Debe incluir o al menos permitir la verificación mediante pruebas.

Ejemplos

Ejemplo 1: Código aceptable

Contexto: Necesitas validar un email en un formulario de registro.

Código generado:

function validarEmail(email) {
  const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return re.test(email);
}

Decisión: Aceptar. Es simple, no tiene dependencias, y maneja el caso básico correctamente. Se podría mejorar la expresión regular, pero es suficiente para el propósito.

Ejemplo 2: Código que requiere corrección

Contexto: Necesitas obtener datos de una API externa.

Código generado:

async function getData() {
  const res = await fetch('https://api.example.com/data');
  return res.json();
}

Decisión: Rechazar y pedir corrección. No maneja errores de red, no tiene timeout, no verifica el código de estado HTTP. Un error de red o un 500 dejaría la aplicación colgada.

💡 Consejo del instructor: Cuando tengas dudas entre aceptar o rechazar, pregúntate: "¿Estaría cómodo desplegando este código a producción sin supervisión adicional?" Si la respuesta es no, entonces necesita al menos una corrección.
⚠ Errores comunes:
  • Aceptar código que "funciona" pero es inseguro: El error más grave. Un código que funciona pero expone datos es peor que un código que no funciona.
  • Rechazar código por estilo personal: Tener preferencias de estilo es válido, pero no debería ser motivo de rechazo si el código sigue los estándares del proyecto.
  • No documentar la decisión: Si rechazas código, explica por qué. Esto ayuda a la IA a mejorar y al equipo a entender el criterio.

Buenas prácticas

  • ✔ Siempre revisa el código generado como si lo hubiera escrito un humano.
  • ✔ Si hay dudas sobre seguridad, busca una segunda opinión.
  • ✔ Documenta el motivo de cada rechazo o corrección.
  • ✔ Establece un estándar mínimo de calidad para tu equipo.
  • ✔ No aceptes código que no entiendas completamente.

Pregunta 1

La IA genera código que cumple el requerimiento pero introduce una dependencia npm nueva no solicitada. ¿Qué haces?

  • a) Lo acepto, la dependencia puede ser útil
  • b) Lo rechazo y pido versión sin dependencias nuevas
  • c) Lo acepto y documento la dependencia para revisión futura
  • d) Elimino la dependencia manualmente sin preguntar

Respuesta: b

Explicación: Las dependencias no solicitadas aumentan la superficie de ataque, el tamaño del proyecto y la deuda técnica. Debes pedir una versión que resuelva el problema sin dependencias externas.

Pregunta 2

El código funciona pero no tiene manejo de errores. El requerimiento no lo mencionaba. ¿Qué haces?

  • a) Lo acepto tal cual, el requerimiento no pedía manejo de errores
  • b) Lo rechazo y pido que añada manejo de errores
  • c) Añado el manejo de errores yo mismo después de aceptarlo
  • d) Lo acepto porque el código funciona en condiciones normales

Respuesta: b

Explicación: El manejo de errores es una responsabilidad implícita de todo código en producción. Aunque el requerimiento no lo mencione, el código debe ser robusto. Pedir a la IA que lo añada es más eficiente que hacerlo tú.

Pregunta 3

La IA genera código correcto pero ineficiente (O(n²) cuando podría ser O(n)). ¿Qué haces?

  • a) Lo acepto, mientras funcione es suficiente
  • b) Pido una optimización a la IA
  • c) Lo rechazo completamente y busco otra solución
  • d) Lo optimizo yo mismo sin avisar

Respuesta: b

Explicación: El código ineficiente puede funcionar en desarrollo pero fallar en producción con datos reales. Pedir una optimización a la IA es rápido y educativo. Solo se rechaza completamente si la ineficiencia es crítica e insalvable.

Pregunta 4

El código tiene un bug que no se detectó en revisión. ¿De quién es la responsabilidad?

  • a) De la IA que generó el código con error
  • b) Del ingeniero que lo integró sin detectar el bug
  • c) De ambos por igual
  • d) Del revisor de código del equipo

Respuesta: b

Explicación: La IA es una herramienta, no un responsable. El ingeniero que integra el código es el último filtro y, por tanto, el responsable de lo que llega a producción. La IA puede equivocarse, pero tú eres quien decide aceptar o rechazar.

Pregunta 5

La IA genera código que usa una librería con una vulnerabilidad conocida. ¿Cuál es la mejor acción?

  • a) Aceptar el código, la vulnerabilidad es conocida y tiene parche
  • b) Rechazar el código y pedir una implementación sin esa librería
  • c) Aceptar el código y actualizar la librería después
  • d) Ignorar la vulnerabilidad porque el código es funcional

Respuesta: b

Explicación: Introducir intencionalmente una dependencia vulnerable es una mala práctica de seguridad. Debes rechazar el código y pedir una alternativa segura, incluso si la vulnerabilidad tiene parche.

Pregunta 6

¿Cuál es la principal ventaja de pedir cambios a la IA en lugar de modificarlos tú mismo?

  • a) La IA siempre genera código mejor que el humano
  • b) Ahorras tiempo y la IA aprende del error para futuras respuestas
  • c) La IA recuerda el contexto del proyecto mejor que tú
  • d) Es más divertido pedir cambios que hacerlos

Respuesta: b

Explicación: Pedir cambios a la IA tiene doble beneficio: ahorras tiempo al no tener que escribir las correcciones manualmente, y la IA incorpora la retroalimentación en su contexto, mejorando respuestas futuras dentro de la misma sesión.

Pregunta 7

Recibes código que es sintácticamente perfecto pero semánticamente incorrecto (hace algo diferente a lo pedido). ¿Qué haces?

  • a) Lo acepto porque la sintaxis es correcta
  • b) Lo rechazo y explico claramente qué debía hacer
  • c) Lo modifico para que haga lo correcto y lo acepto
  • d) Pregunto al equipo si el comportamiento alternativo sirve

Respuesta: b

Explicación: El código sintácticamente perfecto pero semánticamente incorrecto es código incorrecto. Debes rechazarlo y explicar el error para que la IA entienda la diferencia. Aceptarlo introduce bugs silenciosos.

Pregunta 8

¿Cuándo es aceptable usar código generado por IA sin modificaciones?

  • a) Nunca, siempre hay que modificar algo
  • b) Cuando el código pasa todos los puntos de la checklist de revisión
  • c) Solo si el código tiene menos de 10 líneas
  • d) Cuando la IA dice que el código es correcto

Respuesta: b

Explicación: El código generado por IA puede ser aceptado sin modificaciones si pasa la revisión sistemática: cumple el requerimiento, es seguro, maneja errores, es legible, no tiene dependencias innecesarias, y es eficiente.

Resumen

  • Decidir si aceptar o rechazar código generado por IA requiere criterio técnico, no solo verificar que funciona.
  • Los factores clave son: funcionalidad, seguridad, eficiencia, mantenibilidad y dependencias.
  • Un código que "funciona" pero es inseguro debe ser rechazado siempre.
  • Documentar las decisiones de aceptación o rechazo mejora el proceso a largo plazo.
  • La responsabilidad final del código integrado es del ingeniero, no de la IA.
