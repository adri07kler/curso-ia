6.4 Crear una cadena de prompts

Módulo 6 · Ejercicio

Objetivo: Diseñar una cadena de prompts para implementar el proyecto paso a paso.

Introducción

Un prompt aislado puede generar código útil, pero una cadena de prompts bien diseñada puede generar un sistema completo. La clave está en dividir el trabajo en pasos atómicos donde cada prompt recibe el resultado del anterior, construyendo incrementalmente hasta obtener la solución final.

En proyectos reales, intentar generar todo el código en un solo prompt produce resultados de baja calidad: la IA pierde el contexto, omite detalles importantes y genera código inconsistente. Una cadena de prompts divide la complejidad en partes manejables que la IA puede procesar con precisión.

Objetivos de aprendizaje

Al finalizar esta lección el estudiante será capaz de:

  • Dividir un proyecto en pasos atómicos adecuados para prompts individuales
  • Escribir prompts específicos que aprovechen el resultado del paso anterior
  • Diseñar una secuencia lógica de generación que minimice iteraciones
  • Incluir criterios de validación en cada paso de la cadena

Desarrollo del contenido

¿Por qué una cadena de prompts?

Una cadena de prompts (prompt chaining) es una secuencia de instrucciones donde la salida de cada paso se convierte en entrada del siguiente. Esto permite:

  • Dividir la complejidad: Cada prompt resuelve un problema pequeño y específico
  • Mantener el contexto: Cada paso sabe exactamente qué se generó antes
  • Revisar y corregir: Puedes validar paso a paso, no al final
  • Reutilizar pasos: Un buen prompt de modelo de datos sirve para múltiples proyectos

Estructura de una cadena de prompts efectiva

Cada eslabón de la cadena debe incluir:

  1. Contexto necesario: El resultado del paso anterior (o el context packet)
  2. Instrucción específica: Qué debe hacer la IA en este paso exacto
  3. Criterio de validación: Cómo saber si el resultado es correcto
  4. Formato de salida esperado: Cómo debe entregar el resultado para el siguiente paso

Cadena sugerida para el proyecto final

Para nuestra aplicación de tareas, la cadena mínima sugerida es:

Paso 1: Analiza el requerimiento y detecta ambigüedades
Paso 2: Convierte en especificación técnica clara
Paso 3: Propón plan de implementación por pasos
Paso 4: Implementa paso 1 (modelo de datos)
Paso 5: Implementa paso 2 (API/lógica)
Paso 6: Implementa paso 3 (interfaz de usuario)
Paso 7: Genera pruebas para validar
Paso 8: Revisa críticamente el resultado

Ejemplo de un eslabón bien construido

Compara un prompt genérico vs. un prompt dentro de una cadena:

❌ Prompt aislado:

"Crea el modelo de datos para una app de tareas"

✅ Prompt en cadena (Paso 4):

"Basándote en la especificación técnica del Paso 2 y el plan del Paso 3, implementa el modelo de datos de la aplicación. Usa JavaScript vanilla con localStorage. Define las estructuras de: Proyecto, Tarea, Etiqueta. Incluye validación básica y métodos CRUD. El código debe ser funcional y listo para usar en el siguiente paso."

Comparación: Prompt único vs. Cadena de prompts

AspectoPrompt únicoCadena de prompts
Complejidad por promptAlta (todo en uno)Baja (un tema por prompt)
Contexto disponibleSolo el del prompt inicialSe acumula paso a paso
Capacidad de revisiónAl final, todo o nadaPaso a paso, corregir sobre la marcha
Calidad del códigoInconsistente, mezcla patronesConsistente, cada paso refina
Iteraciones necesariasMuchas (regenerar todo)Pocas (regenerar solo el paso fallido)
TrazabilidadBaja (qué salió de dónde)Alta (cada paso documentado)

Ejemplo de cadena de prompts en otros contextos

  • Automatización: (1) Analizar estructura de correos → (2) Diseñar parser → (3) Implementar conexión IMAP → (4) Implementar clasificador → (5) Generar reporte de resultados
  • API REST: (1) Definir endpoints → (2) Diseñar modelos → (3) Implementar controladores → (4) Implementar validación → (5) Escribir tests de integración
  • Script de datos: (1) Analizar formato de entrada → (2) Diseñar pipeline de limpieza → (3) Implementar transformaciones → (4) Implementar exportación → (5) Validar con datos de prueba
💡 Consejo del instructor: No intentes que la IA genere todo en una sola respuesta. Es como pedirle a un arquitecto que construya una casa sin planos. Divide el proyecto en pasos tan pequeños que cada uno pueda validarse independientemente. Si un paso tuviera más de 50 líneas de código, probablemente es demasiado grande — divídelo en dos.

Errores frecuentes

  • Pasos demasiado grandes: Un prompt que intenta generar toda la interfaz de usuario de una sola vez
  • No pasar contexto entre pasos: Cada prompt empieza desde cero y el código se vuelve inconsistente
  • Saltarse la validación intermedia: Llegar al paso 8 y descubrir que el paso 1 estaba mal
  • Orden incorrecto: Implementar UI antes que la lógica de negocio
  • Prompts redundantes: Pedir lo mismo en diferentes pasos porque no se confía en el resultado anterior

Buenas prácticas

  • Diseñar la cadena antes de ejecutarla: Planifica todos los pasos antes de escribir el primer prompt
  • Cada paso debe tener una salida tangible: Código, documento, decisión — algo que se pueda revisar
  • Incluir validación en cada paso: "Verifica que el modelo de datos soporta todos los casos del requerimiento"
  • Numerar los pasos: Para trazabilidad y referencia en correcciones
  • Documentar por qué cada paso es necesario: Ayuda a identificar pasos redundantes o faltantes

Actividad práctica

  1. Divide el proyecto en al menos 6 pasos atómicos
  2. Para cada paso, escribe un prompt específico
  3. Cada prompt debe usar el resultado del anterior

Mini reto

Pide a una IA que diseñe una cadena de prompts para un proyecto diferente (elige uno: generador de menús semanales, analizador de currículums, o chatbot de soporte técnico). Evalúa la cadena propuesta: ¿los pasos son atómicos? ¿Cada prompt incluye criterios de validación? ¿La secuencia es lógica? Mejora la cadena con tus conocimientos y documenta los cambios.

Resumen

  • Una cadena de prompts divide un proyecto grande en pasos pequeños y manejables
  • Cada paso recibe el resultado del anterior como contexto
  • Los pasos deben ser atómicos: cada uno genera una salida revisable
  • La validación intermedia evita arrastrar errores a través de toda la cadena
  • Una cadena bien diseñada produce código más consistente y requiere menos iteraciones
  • Planificar la cadena antes de ejecutarla es tan importante como los prompts mismos
